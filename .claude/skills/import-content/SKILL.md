# import-content

The user will paste content generated by another AI (Gemini, ChatGPT, etc.).
Your job is to detect the content type and call the right injection logic.

---

## Step 1 — Detect content type

Scan the pasted content and classify it as one of:

| Type | Signals |
|---|---|
| **tracks** | Contains JSON with fields like `name`, `emoji`, `desc`, `folder` — one object per track |
| **slides** | Contains blocks with `Type:`, `Eyebrow:`, `Title:`, `Content:` labels |
| **resources** | Contains a list of links/URLs with names and descriptions, grouped by track or topic |
| **mixed** | Contains more than one type above |

If you cannot classify it, ask the user: "This looks like [X] — should I treat it as tracks, slides, or resources?"

---

## Step 2 — Normalize the content

Before injecting, clean up common AI output issues:
- Strip markdown formatting (**, ##, `, etc.) from field values unless it's inside a code block
- Trim extra whitespace and blank lines
- If JSON has trailing commas or single quotes, fix them silently
- If slide content is in prose instead of structured fields, extract the fields yourself
- If a field is missing (e.g. no emoji), use a sensible default (🔹 for missing emoji, `var(--accent)` for missing color)

---

## Step 3 — Route to the right logic

### If type = tracks

Follow the `/add-tracks` logic exactly:
1. Parse each track object
2. Update `trackData` in `docs/codelab.html`
3. Add track cards to `docs/index.html`
4. Add resource section stubs to `docs/resources.html`

Ask the user: "Should I add a 'Bring Your Own Idea' open track at the end?" before writing.

### If type = slides

Follow the `/add-talk-slide` logic for each slide block in sequence:
1. Detect slide type from the `Type:` field (title / principles / bullets / code)
2. Build the correct HTML panel
3. Append to `docs/talk.html` in order

After all slides are added, report the new total slide count.

### If type = resources

For each resource item:
1. Identify which track section it belongs to (match by track name or ask if ambiguous)
2. Read `docs/resources.html`
3. Find the matching `.resource-section` for that track
4. Replace the stub `.resource-item` (or append a new one) with the real content:

```html
<div class="resource-item">
  <span class="resource-name">[name]</span>
  <span class="resource-desc">[description]</span>
  <a class="resource-link" href="[url]" target="_blank">[domain] →</a>
</div>
```

### If type = mixed

Process each type in this order: tracks → slides → resources.
Report each section separately.

---

## Step 4 — Confirm

Print a summary:
- Content type(s) detected
- What was normalized or fixed silently
- Files updated and what changed in each
- Anything that needs manual review (ambiguous track assignments, missing URLs, etc.)

---

## Gemini prompt templates

If the user asks how to generate content for import, share these:

**For tracks:**
```
For each track output a JSON block exactly like this — no extra text:

{
  "name": "Track Name",
  "emoji": "🛍️",
  "desc": "One sentence. What the participant builds.",
  "folder": "01-folder-slug",
  "input": "Example thing a user types in or uploads",
  "color": "var(--accent)"
}

Colors to choose from: var(--accent), var(--accent-2), var(--gold), var(--cream-dim)
Use a different color for each track.

My tracks: [describe your tracks]
```

**For talk slides:**
```
For each slide use exactly this format — no extra text:

Type: [title | principles | bullets | code]
Eyebrow: SHORT LABEL
Title: Slide headline
Content:
- Bullet one
- Bullet two
(or for principles: Name: desc pairs, one per line)

Topic: [your topic]
Number of slides: [N]
Audience: developers at a 2-hour community build jam
```

**For resources:**
```
For each resource use exactly this format:

Track: [track name this belongs to]
Name: Resource display name
Desc: One sentence explaining why it's useful for this track
URL: https://...

My tracks are: [list track names]
Topic/location: [e.g. San Antonio civic data, environmental justice, etc.]
```
